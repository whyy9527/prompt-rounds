# Codex 项目迭代话术

本文档用于指导 Codex / Claude Code / Cursor Agent 等 AI 编码代理持续迭代本项目。
目标不是“功能可用即可”，而是持续把项目打磨得更漂亮、更好用、更符合业务、更容易维护。

---

## 一、使用方式

### 默认原则

- 每一轮迭代前，先阅读：
  - `docs/prd/README.md`
  - `docs/prd/小程序前台_PRD.md`
  - `docs/prd/管理后台_PRD.md`
- 如果本轮只改前台，则重点参考：`小程序前台_PRD.md`
- 如果本轮只改后台，则重点参考：`管理后台_PRD.md`
- 如果涉及前后台联动，则同时参考两份 PRD

### 推荐节奏

- 平时以“总控话术”为主
- 当优化方向跑偏时，再补发“专项话术”
- 当需要约束输出格式时，再附加“输出格式话术”
- 可以连续多轮迭代，但每轮都应优先处理高影响问题，而不是机械重复小修小补

---

## 二、总控话术

先阅读以下业务文档，再开始改代码：

1. `docs/prd/README.md`
2. `docs/prd/小程序前台_PRD.md`
3. `docs/prd/管理后台_PRD.md`

你现在不是单纯写代码，而是作为一个顶级产品工程师持续迭代这个项目。

你的目标不是“功能可用”就结束，而是把项目打磨成：

1. 更漂亮：视觉统一、层级清晰、完成度高；
2. 更好用：操作更直觉、反馈更明确、路径更短；
3. 更符合业务：页面、交互、状态、文案都服务真实业务；
4. 更可维护：代码结构清晰、组件合理、方便后续扩展。

请同时用以下标准审视当前实现：

- Apple / Jobs 风格：克制、简洁、聚焦、直觉、少即是多；
- Jensen Huang 风格：系统性、专业感、整体完成度、工程质量；
- Elon Musk 风格：第一性原理、删繁就简、以结果为导向。

要求：

- 不要只做表面美化，要同时优化信息架构、交互路径、状态反馈、业务闭环；
- 不要只给建议，请直接落到代码修改；
- 主动发现问题，而不是等我指出；
- 优先修改最影响产品质感、易用性和业务承接能力的部分；
- 保持已有关键业务能力不被破坏；
- 修改时尽量保证组件可复用、命名清晰、逻辑内聚；
- 文案不要工程味、后台味、占位味，要像正式上线产品。

本轮请执行：

1. 先快速扫描当前代码与 PRD 的一致性；
2. 找出最值得优先优化的 3~5 项；
3. 直接修改代码；
4. 最后输出：
   - 本轮改了什么
   - 为什么改
   - 对用户体验/业务的提升
   - 与 PRD 的对应关系
   - 下一轮最该继续改什么

## 三、小程序前台专项话术

先阅读：

- `docs/prd/README.md`
- `docs/prd/小程序前台_PRD.md`

本轮只聚焦小程序前台。

请你从“用户体验 + 业务承接 + 产品完成度”三个角度继续迭代前台代码。

重点检查：

- 用户是否一眼明白当前页面是干什么的
- 用户是否清楚下一步该做什么
- 页面是否存在多余步骤、无效信息、弱反馈
- 是否缺少加载态、空态、错误态、成功态
- 文案是否自然、专业、像真正上线产品
- 视觉是否统一，是否存在廉价感、模板感、堆砌感
- 页面是否真正服务核心业务目标，而不是只把功能摆上去

要求：

- 不要只做样式优化，要同时优化交互路径和业务闭环
- 不要只分析，请直接修改代码
- 优先做高影响改动
- 改动必须尽量贴合 `小程序前台_PRD.md` 的目标与约束

输出格式固定为：

1. 问题扫描
2. 本轮改动计划
3. 代码修改
4. 改动说明
5. 与 PRD 的对应关系
6. 下一轮建议

## 四、管理后台专项话术

先阅读：

- `docs/prd/README.md`
- `docs/prd/管理后台_PRD.md`

本轮只聚焦管理后台。

请你从“业务准确性 + 管理效率 + 可维护性”三个角度继续迭代后台代码。

重点检查：

要求：

- 不要只做界面微调，要同时优化后台操作效率和业务逻辑表达
- 不要只给建议，请直接改代码
- 优先处理对业务影响最大、最容易产生误操作的部分
- 改动必须尽量贴合 `管理后台_PRD.md` 的业务规则和目标

输出格式固定为：

1. 问题扫描
2. 本轮改动计划
3. 代码修改
4. 改动说明
5. 与 PRD 的对应关系

## 五、前后台一致性专项话术

先阅读：

- `docs/prd/README.md`
- `docs/prd/小程序前台_PRD.md`
- `docs/prd/管理后台_PRD.md`

本轮请从“前后台一致性”角度检查并迭代代码。

重点关注：

- 前台展示逻辑与后台配置逻辑是否一致
- 前台状态与后台状态流转是否一致
- 后台录入/配置的数据，是否能在前台正确表达业务价值
- 是否存在前台体验合理但后台难维护，或后台能配但前台不好用的问题
- 是否存在命名、状态枚举、字段语义不统一的问题

要求：

- 不要只做局部修补，要优先修复前后台割裂的问题
- 不要只分析，请直接修改代码
- 明确指出哪些问题属于联动缺陷
- 以业务闭环为目标，而不是单看某一端页面

输出时请增加一项：

- 前后台一致性修复说明

## 六、产品质感专项话术

先阅读相关 PRD，再继续优化当前实现。

请把这个项目继续往“高级、克制、可信赖”的方向打磨。

我不要廉价感，不要页面元素堆满，不要颜色乱飞，不要低级运营风。

请重点优化：

- 视觉层级
- 留白和密度
- 字体大小与权重关系
- 卡片、按钮、输入框的一致性
- 颜色系统与主辅色逻辑
- 页面第一眼的品牌感和完成度

标准：

- 像一个认真做体验的成熟产品
- 像 Apple 式的简洁与专注
- 像高质量科技产品，而不是模板站

请直接修改代码，不要只提建议。

## 七、易用性专项话术

先阅读相关 PRD，再从“更好用”角度继续优化。

请把自己当成第一次使用这个产品的新用户，检查：

- 是否一眼知道下一步该做什么
- 是否有多余步骤
- 是否有不明确的按钮、状态、反馈
- 是否有让人犹豫或困惑的地方
- 是否缺少加载、空态、错误态、成功态
- 是否存在点击后没反馈、反馈太弱、反馈太晚的问题

请优先优化那些会影响：

- 首次使用成功率
- 关键流程完成率
- 用户信任感
- 操作流畅感

直接改代码，并说明你消除了哪些摩擦点。

---

## 八、业务闭环专项话术

先阅读相关 PRD，再从业务目标出发继续迭代。

请检查当前设计是否真正服务业务：

- 页面是否围绕核心转化目标展开
- 用户是否知道为什么要用、用了有什么价值
- 是否缺少引导用户继续下一步的设计
- 是否缺少关键数据展示、结果反馈、转化入口
- 是否有与业务目标无关的冗余内容
- 是否能让用户形成“进入-理解-操作-获得结果-继续使用”的闭环

请删除无效设计，强化核心路径，直接修改代码。
输出时请明确说明：哪些改动提升了业务承接能力。

---

## 九、工程质量专项话术

在保证页面效果继续提升的同时，请同步提升工程质量。

请检查并优化：

- 组件拆分是否合理
- 是否有重复代码
- 状态管理是否混乱
- 命名是否表达业务含义
- 样式是否可复用
- 页面逻辑是否过于耦合
- 是否方便后续继续扩展功能

原则：

- 不为了过度抽象而抽象
- 但也不要把演示代码继续堆成生产代码
- 让这个项目更像一个可持续迭代的正式项目

请直接改代码，并说明本轮做了哪些“长期有价值”的整理。

## 十、输出格式约束话术

这次不要只分析，请直接进入修改。

输出格式必须固定为：

1. 问题扫描

- 列出你发现的最关键问题（不超过 5 条）

1. 本轮改动计划

- 只选高影响的 3~5 项，不要贪多

1. 代码修改

- 直接落代码
- 如涉及多个文件，逐文件修改
- 不要只给伪代码

1. 改动说明

- 改了什么
- 为什么改
- 对体验 / 业务 / 维护性的提升

1. 与 PRD 的对应关系

- 说明本轮改动对应了哪些业务目标、页面要求或规则

1. 下一轮建议

- 给出下一轮最值得继续优化的 3 项

要求：

- 每轮都比上一轮更接近可上线产品
- 不要重复已经完成的优化
- 不要为了改而改
- 优先做高收益改动

## 十三、推荐使用策略

### 方案一：稳定迭代

- 第 1 轮：总控话术 + 输出格式约束
- 第 2 轮：总控话术
- 第 3 轮：总控话术
- 第 4 轮：按偏差补一个专项话术
- 第 5 轮以后：继续总控，发现跑偏再补专项

### 方案二：高压迭代

- 每一轮先发短版总控
- 再根据当前问题补一个专项
- 每隔几轮加一次输出格式约束

### 方案三：前后台分治

- 前台阶段：总控 + 小程序前台专项 + 产品质感 / 易用性 / 业务闭环
- 后台阶段：总控 + 管理后台专项 + 业务闭环 / 工程质量
- 联调阶段：总控 + 前后台一致性专项

---

## 十四、建议

默认用总控话术做底座。
发现哪里偏，就补哪个专项，不用每一轮都发全家桶。
目标是让代理持续收敛，而不是机械重复。

最终标准不是“改了很多”，而是：

- 页面更成熟
- 体验更顺
- 业务更通
- 结构更稳
